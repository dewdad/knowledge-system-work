# KSW Init Flow

> Loaded when: the user runs `/init` and the workspace contains neither `ksw.yaml` nor `.ksw-link.yaml`. Also loaded when re-running `/init` after upgrading the skill (to refresh templates).
>
> Two modes are supported: **Hub** (full system) and **Satellite** (lightweight bridge to an existing hub). The user picks at Step 0.

Cross-cutting references:
- Platform CLI commands: [PLATFORM-OPS.md](PLATFORM-OPS.md).
- State machine, label catalogue: [COORDINATION.md](COORDINATION.md).
- Hook templates installed by Steps 9–10 (hub) and 6–7 (satellite): [`reference/hooks/`](reference/hooks/).

---

## Prerequisites (both modes)

```bash
# Git initialized?
git rev-parse --is-inside-work-tree

# Platform CLI available? (hub may also run in `local` mode without one)
glab auth status   # GitLab
gh auth status     # GitHub

# Required by generated hooks/workflows
yq --version
jq --version

# Optional integrations
markitdown --version  # pip install markitdown[all]
yt-dlp --version      # YouTube sources
```

If none of these are installed, surface a single clear list of missing tools and stop.

## Step 0 — Choose Mode

Ask the user:

> **What role should this workspace play?**
> 1. **Hub** — Central KSW repository (issue board, wiki, domains, synthesis)
> 2. **Satellite** — Project workspace that bridges to an existing KSW hub

Pre-flight checks:
- Workspace already contains `ksw.yaml` → warn `KSW hub already initialized here.` and ask whether to abort or refresh.
- Workspace already contains `.ksw-link.yaml` → warn `Satellite already linked here.` and ask whether to abort or refresh.

Record the chosen mode and proceed to **Step 1**.

## Step 1 — Authenticate & Select Repository

### 1a. Platform Authentication

```bash
glab auth status 2>&1   # Check GitLab
gh auth status 2>&1     # Check GitHub
```

- Neither authenticated:
  - **Hub** may continue in `local` mode (filesystem queue, no remote).
  - **Satellite** is unavailable. Prompt: `Run \`glab auth login\` or \`gh auth login\` first.`
- Both authenticated → ask which platform hosts (or will host) the KSW hub.
- Record the platform choice.

### 1b. Repository Selection

**Hub mode:**

```bash
# List user's repos
glab repo list --mine          # GitLab
gh repo list --limit 20        # GitHub

# Use existing or create new?
glab project create --name "<name>" --description "Knowledge Work System"
gh repo create "<name>" --description "Knowledge Work System" --private
```

If the user picks an existing repo, verify it is empty or already has a KSW structure (`ksw.yaml` + `domains/` + `wiki/`). Refuse to overwrite an unrelated repo.

**Satellite mode:**

```bash
# List user's repos — filter for likely KSW hubs
glab repo list --mine          # GitLab
gh repo list --limit 20        # GitHub

# Verify the selected repo is a KSW hub:
glab label list -R <selected_path> | grep "state:inbox"
gh label list -R <selected_path> | grep "state:inbox"
```

If no KSW labels are present, warn: `Selected repo doesn't appear to be a KSW hub. Run /init as hub there first.` and stop.

**Satellite-on-local-hub guard.** If the user picked Satellite **and** the chosen hub repo has `instance.platform: local` (no remote, ksw.yaml only on disk), abort with:

> Satellite mode requires a remote-accessible hub (GitLab or GitHub). The selected hub is local-only.
> Either re-init the hub with a remote platform, or use the hub directly without a satellite.

Record `project_path` and proceed:
- **Hub** → [Hub Init Flow](#hub-init-flow) below
- **Satellite** → [Satellite Init Flow](#satellite-init-flow) below

## Hub Init Flow

Continues from Step 1 when `mode = hub`.

### Step 2 — Detect Platform

Already determined in Step 1. Stored as one of:
- `gitlab` → primary supported target (`glab` CLI)
- `github` → documented command surface (`gh` CLI); verify generated hooks before unattended use
- `local` → filesystem-only queue (hub only; no satellites, no remote automation)

### Step 3 — Create Directory Structure

```
<project_root>/
├── .ksw/
│   ├── queue/               ← Local task queue (platform:local)
│   │   ├── inbox/
│   │   ├── ready/
│   │   ├── wip/
│   │   ├── done/
│   │   └── blocked/
│   ├── workflows/           ← Generated workflow docs (agent reads these)
│   ├── state/
│   └── cache/
├── domains/
├── wiki/
│   ├── _meta/
│   │   └── briefs/
│   ├── concepts/
│   ├── entities/
│   ├── decisions/
│   ├── projects/
│   ├── synthesis/
│   └── _graph/
├── raw/
│   ├── papers/
│   ├── transcripts/
│   ├── exports/
│   └── screenshots/
├── ksw.yaml
└── AGENTS.md
```

Create all directories with `mkdir -p` (bash) or `New-Item -Force` (PowerShell).

### Step 4 — Generate `ksw.yaml`

```yaml
# ksw.yaml — Knowledge Work System Configuration
# Generated by /init (hub mode) on <TIMESTAMP>

ksw:
  skill_version: "0.6.0"      # SKILL.md version that generated this file
  config_version: 1           # Bumped only on breaking schema changes

instance:
  name: "<project_name>"
  owner: "<detected_username>"
  platform: "<gitlab|github|local>"
  project_path: "<owner/repo or local path>"
  mode: "hub"

identity:
  timezone: "<detected_or_ask>"
  locale: "en-US"

domains: []

satellites: []          # Populated when satellites register via /init satellite

coordination:
  mode: "<solo|team>"
  default_branch: main
  mr_required: true
  stale_wip_timeout_minutes: 30
  max_parallel_agents: 3

scheduling:
  source_pull: "0 3 * * *"
  wiki_lint: "0 4 * * *"
  morning_brief: "0 7 * * *"
  weekly_review: "0 20 * * 0"

wiki:
  format: obsidian
  wikilinks: true
  frontmatter: true
  graph: true

tools:
  markitdown: auto    # auto | path | disabled
```

The `ksw.skill_version` field lets future versions detect mismatched configs and prompt the user to migrate. `config_version` only increments when the YAML structure breaks compatibility — additive fields do not bump it.

### Step 5 — Generate `AGENTS.md`

Write `AGENTS.md` to project root containing:

1. **System Overview** — `This project uses KSW. Flow: Sources → Issues/Queue → Wiki → Synthesis.`
2. **Quick Start** — Read `ksw.yaml`, check for work, claim → branch → do.
3. **Domain map** — list of configured domains with one-line descriptions.
4. **Pointer to generated workflow docs** — `.ksw/workflows/`.
5. **Coordination summary** — mode (solo/team), branch convention, claim/release rules. Link to [COORDINATION.md](COORDINATION.md).

### Step 6 — Create `.gitignore` entries

Append to `.gitignore`:

```
# KSW internals
.ksw/cache/
.ksw/state/
secrets/                # Source pull credentials — NEVER COMMIT
```

The `secrets/` directory holds per-source credential stubs created by `/add-source` (token/basic-auth/OAuth shapes — see [HUB-COMMANDS.md § /add-source](HUB-COMMANDS.md#add-source-domain-type-id) Step 4).

### Step 7 — Platform Setup

Create every label in [COORDINATION.md § Coordination labels](COORDINATION.md#coordination-labels-canonical-list). The canonical YAML form is at [`reference/coordination/labels.yaml`](reference/coordination/labels.yaml). Use the platform commands from [PLATFORM-OPS.md § Label creation](PLATFORM-OPS.md#label-creation-hub-init).

For local mode, create `.ksw/queue/<state>/` subdirectories (already done in Step 3) instead of platform labels.

### Step 8 — Generate Workflow Documentation

Create `.ksw/workflows/` and copy/render the canonical workflow files from [`reference/workflows/`](reference/workflows/). Each generated file follows: **Trigger → Steps → Error Handling → Commit convention**, with platform-specific commands resolved per [PLATFORM-OPS.md](PLATFORM-OPS.md).

| File | Purpose | Source |
|------|---------|--------|
| `source-pull.md` | Pull new items from sources | [`reference/workflows/source-pull/SKILL.md`](reference/workflows/source-pull/SKILL.md) |
| `issue-triage.md` | Classify & prioritize inbox | [`reference/workflows/issue-triage/SKILL.md`](reference/workflows/issue-triage/SKILL.md) |
| `wiki-ingest.md` | Raw → wiki pages | [`reference/workflows/wiki-ingest/SKILL.md`](reference/workflows/wiki-ingest/SKILL.md) |
| `wiki-synthesize.md` | Cross-domain patterns | [`reference/workflows/wiki-synthesize/SKILL.md`](reference/workflows/wiki-synthesize/SKILL.md) |
| `graph-build.md` | Wikilink graph index | (zero-LLM; algorithm in [WORKFLOWS.md § Graph Build](WORKFLOWS.md#graph-build-graph-build)) |
| `domain-review.md` | Weekly domain health | [`reference/workflows/domain-review/SKILL.md`](reference/workflows/domain-review/SKILL.md) |
| `issue-to-wiki.md` | Closed issue → ADR | [`reference/workflows/issue-to-wiki/SKILL.md`](reference/workflows/issue-to-wiki/SKILL.md) |
| `wiki-to-issue.md` | Wiki gap → issue | [`reference/workflows/wiki-to-issue/SKILL.md`](reference/workflows/wiki-to-issue/SKILL.md) |
| `morning-brief.md` | Daily summary | [`reference/workflows/morning-brief/SKILL.md`](reference/workflows/morning-brief/SKILL.md) |
| `hub-hooks.md` | Hook install/maintenance | [`reference/hooks/hub/`](reference/hooks/hub/) — procedure in Steps 9–10 below |
| `init-smoke-test.md` | Post-init verification | [`reference/workflows/init-smoke-test/SKILL.md`](reference/workflows/init-smoke-test/SKILL.md) |

### Step 9 — Install Agent Hooks

Detect AI tools in workspace (`.opencode/`, `.claude/`, `.cursor/`) and install lifecycle hooks.

- **OpenCode** → `.opencode/hooks/ksw-hub.yaml` (session start: inbox count, stale WIP, brief status; session end: prompt graph-build, state verification, overdue pulls).
- **Claude Code** → append to `CLAUDE.md` (equivalent on-start / on-wrap-up sections).
- **No tools detected** → skip; the AGENTS.md from Step 5 already provides equivalent guidance.

Templates: [`reference/hooks/hub/agents/`](reference/hooks/hub/agents/). Generated docs land in `.ksw/workflows/hub-hooks.md`.

### Step 10 — Install Git Hooks

Write to `.git/hooks/` (or append between guard markers if hooks already exist):

| Hook | Purpose | Trigger |
|------|---------|---------|
| `post-commit` | Batched progress comment on issue (every Nth commit on `ksw/<ID>-*` branch) | Commit on issue branch |
| `post-merge` | Transition issue `state:wip` → `state:review` | Branch merge to default branch |
| `pre-push` | Warn on broken wikilinks (non-blocking) | Push |
| `post-checkout` | Display issue title/description | Switch to `ksw/<ID>-*` branch |

Coexistence: existing hooks (husky, lint-staged, …) are preserved by appending between `# [KSW-HUB-HOOK-START]` / `# [KSW-HUB-HOOK-END]` guard markers.

```bash
chmod +x .git/hooks/*
```

Templates: [`reference/hooks/hub/git/`](reference/hooks/hub/git/).

### Step 11 — Smoke Test & Completion

Run the Hub section of [`reference/workflows/init-smoke-test/SKILL.md`](reference/workflows/init-smoke-test/SKILL.md) — including the **directory-install check**: try to read at least one expected hook template path (e.g. `reference/hooks/hub/git/post-commit`). If it fails to resolve, surface:

> Skill installed without `reference/` siblings — re-install as a directory.
> See [INSTALL.md](INSTALL.md).

Then output:

```
KSW Hub initialized successfully.

Platform: <platform>
Mode: <coordination_mode>
Config: ksw.yaml (skill_version 0.6.0)
Workflow docs: .ksw/workflows/

Installed:
  ✓ ksw.yaml (configuration)
  ✓ AGENTS.md (agent instructions)
  ✓ .ksw/workflows/ (11 workflow docs)
  ✓ Agent hooks (<detected_tool> lifecycle)
  ✓ Git hooks (post-commit, post-merge, pre-push, post-checkout)

Next steps:
  1. Add a domain:    /add-domain health
  2. Add a source:    /add-source health rss examine-research
  3. Pull sources:    /pull
  4. Check status:    /status
```

## Satellite Init Flow

Continues from Step 1 when `mode = satellite`. Installs persistent mechanisms so all future AI sessions and git operations automatically bridge to the hub.

### Step 2 — Gather Satellite Config

1. **Workspace name** — default to current directory name; ask user to confirm/override.
2. **Default domain** — list domains from hub labels:
   ```bash
   glab label list -R <hub_path> | grep "domain:"    # GitLab
   gh   label list -R <hub_path> | grep "domain:"    # GitHub
   ```
   Ask the user to pick a primary domain (pre-fills new issues).
3. **Satellite label** — create `satellite:<workspace_name>` on the hub if absent:
   ```bash
   glab label create "satellite:<name>" --color "#1D76DB" --description "Routed to <name> workspace" -R <hub_path>
   gh   label create "satellite:<name>" --color "1D76DB"  --description "Routed to <name> workspace" -R <hub_path>
   ```

### Step 3 — Write `.ksw-link.yaml`

```yaml
# .ksw-link.yaml — Satellite bridge config
# Generated by /init (satellite mode) on <TIMESTAMP>

ksw:
  skill_version: "0.6.0"
  config_version: 1

hub:
  platform: "<gitlab|github>"
  project_path: "<owner/ksw-hub>"
  default_branch: "main"          # Read from hub's ksw.yaml#coordination.default_branch

identity:
  workspace_name: "<workspace_name>"
  workspace_path: "<absolute_path>"
  satellite_label: "satellite:<workspace_name>"

preferences:
  default_domain: "<selected_domain>"
  progress_interval: 5            # Comment every N commits on issue branch
  auto_issue_ref: true            # prepare-commit-msg appends (KSW #ID)

active_claims: []                 # Issue IDs currently being worked here
```

The `hub.default_branch` field is consulted by satellite git hooks instead of hard-coding `main`. If reading the hub's `ksw.yaml` fails during init, default to `main` and warn the user.

### Step 4 — Update `.gitignore`

Append:

```
# KSW satellite (personal config, not shared)
.ksw-link.yaml
```

### Step 5 — Augment `AGENTS.md`

Append the KSW Satellite section to the workspace `AGENTS.md` (create the file if missing). This gives agents judgment heuristics for:
- Session awareness (active claims, routed work)
- Decision detection (offer to record as wiki decision)
- Knowledge extraction (offer to contribute to hub wiki via `/sat contribute`)
- Issue creation (with satellite + domain labels)
- Context linking (KSW #ID references in commits)
- What NOT to do — no trivial artifacts, no duplicates, always ask first.

Template: [`reference/hooks/satellite/agents-md-section.md`](reference/hooks/satellite/agents-md-section.md).

### Step 6 — Install Agent Hooks

Detect AI tools (`.opencode/`, `.claude/`, `.cursor/`) and install lifecycle hooks:

- **OpenCode** → `.opencode/hooks/ksw-satellite.yaml` (session start: hub, active claims, routed work; session end: prompt progress logging, decision contribution).
- **Claude Code** → append start/wrap-up sections to `CLAUDE.md`.
- **No tools detected** → skip; the AGENTS.md section from Step 5 covers it.

Templates: [`reference/hooks/satellite/agents/`](reference/hooks/satellite/agents/).

### Step 7 — Install Git Hooks

Write to `.git/hooks/` (or append between guard markers if hooks exist):

| Hook | Purpose | Trigger |
|------|---------|---------|
| `post-commit` | Comment on hub issue — explicit `KSW #ID` refs or batch every N commits on `ksw/<ID>-*` branches | Commit |
| `post-merge` | Transition hub issue `state:wip` → `state:review`, remove from `active_claims` | Branch merge to hub's default branch |
| `prepare-commit-msg` | Auto-append `(KSW #ID)` on `ksw/<ID>-*` branches | Commit message creation |

Coexistence: append between `# [KSW-SAT-HOOK-START]` / `# [KSW-SAT-HOOK-END]` guard markers if hooks already exist.

```bash
chmod +x .git/hooks/*
```

Templates: [`reference/hooks/satellite/git/`](reference/hooks/satellite/git/).

### Step 8 — Register with Hub

Register the satellite in the hub's `ksw.yaml` (via API or sparse checkout):

```bash
TMPDIR=$(mktemp -d "${TMPDIR:-/tmp}/ksw-register.XXXXXX")
trap 'rm -rf "$TMPDIR"' EXIT
git clone --no-checkout --depth 1 <hub_url> "$TMPDIR"
cd "$TMPDIR"
git sparse-checkout set ksw.yaml
git checkout
yq -i '.satellites += [{"name": "<workspace_name>", "label": "satellite:<workspace_name>", "default_domain": "<domain>", "registered_at": "<ISO8601>"}]' ksw.yaml
git add ksw.yaml
git commit -m "feat: register satellite <workspace_name>"
git push
```

If the push fails (permissions, protected branch): report the failure and continue. Registration is informational — the satellite works without it, but the user should know hub inventory is incomplete.

### Step 9 — Verify & Complete

1. Run the Satellite section of [`reference/workflows/init-smoke-test/SKILL.md`](reference/workflows/init-smoke-test/SKILL.md), including the **directory-install check** (read `reference/hooks/satellite/git/post-commit`). On failure, surface the same `re-install as a directory` message as the hub Step 11.
2. Dry-run connectivity test:
   ```bash
   glab issue list -R <hub> --label "satellite:<name>" --per-page 1
   ```
3. Display summary:

```
KSW Satellite initialized.

Hub: <project_path> (<platform>)
Workspace: <workspace_name>
Label: satellite:<workspace_name>
Domain: <default_domain>
Skill version: 0.6.0

Installed:
  ✓ .ksw-link.yaml (config)
  ✓ AGENTS.md (KSW Satellite section appended)
  ✓ <agent_tool> hooks (session lifecycle)
  ✓ .git/hooks/post-commit (progress tracking)
  ✓ .git/hooks/post-merge (completion detection)
  ✓ .git/hooks/prepare-commit-msg (issue ref injection)

This workspace is now tracked by KSW. All future AI sessions
will have KSW context, and git operations will report progress
to hub issues automatically.

On-demand: /sat board, /sat claim, /sat done, /sat new, /sat contribute
```

## Schemas

These schemas are normative. The agent uses them when generating or validating config files during `/init`, `/add-domain`, and `/add-source`.

### `ksw.yaml`

```yaml
ksw:
  skill_version: string  # SKILL.md version that wrote this file
  config_version: integer (default: 1)

instance:
  name: string
  owner: string
  platform: enum[gitlab, github, local]
  project_path: string
  mode: enum[hub, satellite]   # Determined at /init Step 0

identity:
  timezone: string       # IANA timezone
  locale: string         # xx-YY

domains: string[]

satellites: []           # Hub-only: registered satellite workspaces
  # - name: "frontend-app"
  #   label: "satellite:frontend-app"
  #   default_domain: "engineering"
  #   registered_at: "<ISO8601>"

coordination:
  mode: enum[solo, team]
  default_branch: string (default: main)
  mr_required: boolean (default: true, team only)
  stale_wip_timeout_minutes: integer (10-1440, default: 30)
  max_parallel_agents: integer (1-10, default: 3)

scheduling:              # cron expressions (advisory — agent checks if due)
  source_pull: string
  wiki_lint: string
  morning_brief: string
  weekly_review: string

wiki:
  format: enum[obsidian, markdown]
  wikilinks: boolean (default: true)
  frontmatter: boolean (default: true)
  graph: boolean (default: true)

tools:
  markitdown: enum[auto, disabled] | string
```

### `.ksw-link.yaml` (satellite only)

```yaml
ksw:
  skill_version: string
  config_version: integer (default: 1)

hub:
  platform: enum[gitlab, github]
  project_path: string
  default_branch: string (default: main)

identity:
  workspace_name: string
  workspace_path: string
  satellite_label: string        # "satellite:<workspace_name>"

preferences:
  default_domain: string
  progress_interval: integer (1-20, default: 5)
  auto_issue_ref: boolean (default: true)

active_claims: integer[]         # Issue IDs currently claimed
```

## Compatibility

`ksw.skill_version` is informational in 0.6.0 — KSW reads it and warns on a major-version mismatch but does not refuse to operate. `ksw.config_version` is reserved: it stays at `1` for all 0.6.x releases. Future breaking changes to the config schema will bump `config_version` and ship an explicit migration path.
