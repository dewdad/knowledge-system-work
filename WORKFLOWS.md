# KSW Workflow Index

> Loaded when: a workflow command is invoked (`/pull`, `/triage`, `/ingest`, `/synthesize`, `/review`, `/brief`, `/graph-build`) or a generated workflow doc is consulted. This file is a router/index — it summarizes each workflow and points to the authoritative detail in [`reference/workflows/`](reference/workflows/) (and the generated copy in `.ksw/workflows/`).

Every summary below follows: **Trigger → Flow → Failure rule → Detailed reference.** Detailed steps belong in `reference/workflows/<name>/SKILL.md`. SKILL.md never inlines them.

---

## Source Pull (`/pull [domain]`)

Pull new items from domain sources into the inbox.

- **Trigger**: scheduled (`scheduling.source_pull` cron), manual, or after adding a new source.
- **Flow**: For each domain → read `domains/<d>/sources.yaml` → check if due (`.state/pulls.json` vs schedule) → pull by source type (RSS: fetch+parse XML; YouTube: `yt-dlp --flat-playlist`; API: HTTP GET; Git: API commits/releases) → derive stable `source_item_id` (source id + canonical URL/external id hash) → skip if already seen or issue exists → create inbox items with labels `state:inbox, domain:<d>, type:source-item` → update pull state → commit.
- **Failure rule**: A source unreachable → log in `pulls.json`, increment `failures` counter; after 5 consecutive failures create a `type:maintenance` issue. Never block other sources.
- **Detail**: [`reference/workflows/source-pull/SKILL.md`](reference/workflows/source-pull/SKILL.md).

## Issue Triage (`/triage`)

Classify and prioritize all `state:inbox` items.

- **Trigger**: after source pull, when new inbox items appear, or manual.
- **Flow**: List `state:inbox` items → for each, infer **domain** (keyword match against configured domains), **type** (action verb→`task` | question→`research` | choose/decide→`decision` | cleanup→`maintenance`), **priority** (urgency→`P0` | deadline→`P1` | default→`P2` | exploratory→`P3`) → apply labels, transition `state:inbox` → `state:ready`, add a triage note explaining the classification.
- **Failure rule**: ambiguous items get `needs:clarification` and stay in inbox. Source items default to `P2` if priority can't be inferred.
- **Detail**: [`reference/workflows/issue-triage/SKILL.md`](reference/workflows/issue-triage/SKILL.md).

## Wiki Ingest (`/ingest <path>`)

Process raw material into structured wiki pages.

- **Trigger**: file dropped in `raw/`, manual, or after research.
- **Flow**: Convert non-markdown via `markitdown` if available → extract knowledge units → classify (concept | entity | decision | project | synthesis) → resolve against existing wiki (merge or create) → write with Obsidian frontmatter and `[[wikilinks]]` → rebuild graph → commit.
- **Failure rule**: One concept per page. Every page has stable `id`, `aliases`, `domain`, `created`, `updated`, `sources`. Inline source references for material-derived claims. No duplication.
- **Detail**: [`reference/workflows/wiki-ingest/SKILL.md`](reference/workflows/wiki-ingest/SKILL.md).

## Wiki Synthesize (`/synthesize`)

Cross-domain pattern detection and insight generation.

- **Trigger**: weekly schedule, after 10+ new wiki pages, or manual.
- **Flow**: Load graph index (fallback: grep) → detect patterns (concepts in 3+ domains, orphans, contradictions, inconsistent tags) → generate synthesis pages in `wiki/synthesis/` → update `_meta/_insights.md` → create issues for actionable findings → rebuild graph.
- **Failure rule**: Graph is an optimization, never a hard dependency. If `graph.json` is missing, fall back to grep across `wiki/`.
- **Detail**: [`reference/workflows/wiki-synthesize/SKILL.md`](reference/workflows/wiki-synthesize/SKILL.md).

## Graph Build (`/graph-build`)

Rebuild the wikilink graph index. Zero-LLM, deterministic.

- **Trigger**: after ingest, after synthesize, or manual.
- **Flow**: Scan `wiki/**/*.md` → extract frontmatter + `[[wikilinks]]` → build adjacency list → compute stats (incoming/outgoing counts, orphans, most-connected top 10, domain/category counts) → write `wiki/_graph/graph.json` and `wiki/_graph/orphans.md` → report summary.
- **Failure rule**: Pure file IO; failures are filesystem-level. Never silently overwrites — always emits both files together.
- **Detail**: generated by `/init` into `.ksw/workflows/graph-build.md`. The algorithm is fully specified in this section — there is no separate `reference/workflows/graph-build/` because the workflow has no LLM steps to encode.

## Domain Review (`/review <domain>`)

Weekly health check on a single domain.

- **Trigger**: scheduled per `domain.yaml#review_schedule`, or manual.
- **Flow**: source health (stale/broken in `pulls.json`) → issue health (count by state, stalled items) → wiki coverage (pages, recency) → generate report → create issues for problems found.
- **Failure rule**: Read-only beyond report and issue creation. Never closes user issues.
- **Detail**: [`reference/workflows/domain-review/SKILL.md`](reference/workflows/domain-review/SKILL.md).

## Morning Brief (`/brief`)

Daily system state summary.

- **Trigger**: scheduled daily, or manual.
- **Flow**: gather due items, in-flight, completed (24h), blocked, queue depth → source status (overdue pulls) → wiki activity (24h modifications) → compose brief (≤50 lines) → write to `wiki/_meta/briefs/YYYY-MM-DD.md`.
- **Failure rule**: never overwrites a same-day brief — appends `-1`, `-2`, … if needed. Never blocks on a missing graph index.
- **Detail**: [`reference/workflows/morning-brief/SKILL.md`](reference/workflows/morning-brief/SKILL.md).

## Issue → Wiki (automatic)

Closed issues with decisions or significant knowledge produce wiki decision records.

- **Trigger**: issue with `type:decision` closed, or significant `type:task` closed.
- **Flow**: Read closed issue → create ADR-style wiki page (Context / Decision / Rationale / Consequences) → cross-link bidirectionally with `[[wikilinks]]` → commit.
- **Detail**: [`reference/workflows/issue-to-wiki/SKILL.md`](reference/workflows/issue-to-wiki/SKILL.md).

## Wiki → Issue (automatic)

Wiki gaps and contradictions produce actionable issues.

- **Trigger**: synthesis reveals gap, contradiction found, review identifies missing coverage, or `wiki/_graph/orphans.md` lists a long-standing orphan.
- **Flow**: Identify actionable insight → classify (gap → `type:research`, contradiction → `type:decision`, missing → `type:task`) → create issue with wiki context → update wiki page with the new issue link.
- **Detail**: [`reference/workflows/wiki-to-issue/SKILL.md`](reference/workflows/wiki-to-issue/SKILL.md).

## Known limitation: cross-host pull lock

The current `/pull` workflow uses a local-filesystem lockfile to prevent two concurrent pulls of the same source on the same host. It does not coordinate across satellites pulling the same hub-defined source from different machines. A platform-agnostic `pulling:<domain>` label with TTL is planned for a future minor; until then, schedule pulls in one place per source.
